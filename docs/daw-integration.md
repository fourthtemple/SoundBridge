# Web DAW Integration

SoundBridge should feel like an AudioNode plus a plugin-management API.

## Minimal Host Flow

1. Create `AudioContext`.
2. Create `SoundBridgeClient` with `ws://127.0.0.1:47370/bridge`.
3. Call `connect()`, `hello()`, and `pair()`.
4. Call `scanPlugins()` or `listPlugins()`.
5. Create a plugin instance with the current sample rate and block size.
6. Create `SoundBridgeAudioNode`.
7. Connect source nodes into the bridge node and connect the bridge node to the destination.
8. For instruments, send notes or MIDI clips through `sendMidiEvents()`; for VST3 note expression, include explicit bounded `noteId` values.
9. Apply bounded listed presets with `setPreset()` when the selected plugin exposes preset metadata.
10. Check the plugin's `editorKinds`, then open a bounded generic parameter editor with `openEditor()` or a configured native editor broker when both the plugin and daemon advertise it.
11. Close editor sessions with `closeEditor()` when the UI tab or panel is done.
12. Store `getState()` output in the DAW project.
13. Restore using `setState()` when reopening the project.

## WebAudioModules Compatibility

The first compatibility target is an adapter that exposes:

- descriptor metadata from `listPlugins()`
- format-aware plugin identity for VST3, AU, LV2, and mock plugins
- an AudioNode-like instance backed by `SoundBridgeAudioNode`
- normalized parameter automation
- bounded preset snapshot application
- opaque state serialization
- generic parameter UI fallback

The adapter should not require hosts to adopt SoundBridge-specific UI primitives.

## Host Responsibilities

Hosts should:

- clearly show when audio is being routed through a local daemon
- use `createInstance().layout` or `getLayout()` when sizing host nodes and meters
- store opaque plugin state without editing it
- account for `getLatency()` in timeline scheduling and monitoring UI; `SoundBridgeLiveEffectRack.refreshLatency()`, `SoundBridgeAudioNode.refreshLatency()`, `createLivePerformanceAudioNodePolicy()`, `createLivePerformanceAudioNodeCalibrationWindow()`, `calibrateLivePerformanceAudioNodePolicy()`, and render-response diagnostics keep plugin, transport, and total reported latency in host-readable sample and millisecond health
- call `SoundBridgeAudioNode.setBypassed(true)` for Web Audio graph emergency fail-dry paths instead of destroying and recreating the plugin instance during a live set; the worklet outputs dry input and discards queued or pending wet responses from before the bypass transition, and `setBypassed(false)` is the explicit retry after transport-pressure, render-pressure, or ordinary audio-error auto-bypass
- treat `SoundBridgeAudioNode.health.unhealthyReason === "process-timeout"` as a native render deadline/quarantine signal from `render_timeout` or `render_quarantined`; the AudioNode emits `process-timeout`, `process-timeout-tripped`, and `process-timeout-auto-bypassed`, `retry()` and `setBypassed(false)` will not clear that quarantined state, and hosts should create a fresh plugin instance/node or use `SoundBridgeLiveEffectRack` when they want bounded timeout recreation
- read `SoundBridgeAudioNode.health` and listen for `latencychange`, `fallback-output`, `response-deadline-missed`, `transport-pressure`, `transport-pressure-auto-bypassed`, `render-budget-exceeded`, `render-budget-tripped`, `render-budget-auto-bypassed`, `process-timeout`, `process-timeout-tripped`, `process-timeout-auto-bypassed`, `audio-error-auto-bypassed`, and `audio-error` when using the Web Audio graph path; health includes bypass state, transport pressure, transport-pressure auto-bypass, render pressure, render-budget auto-bypass, process-timeout quarantine, audio errors, audio-error auto-bypass, plugin/transport/reported latency in samples and milliseconds, render-response plugin latency changes, adaptive latency counters, fallback output counters and reasons, deadline misses, stale output blocks, dropped input blocks, underruns, shared-audio drops, shared-worker saturation, response jitter against the configured threshold, and the latest pressure/latency reasons; shared-ring input pressure and response jitter can raise bounded adaptive latency, `response-jitter` and `shared-transport-saturation` can appear as transport-pressure reasons, and `transportPressureAutoBypassReasons` lets hosts observe soft pressure while failing dry only for selected hard reasons
- send bounded `processAudioBlock.transport` context when the host knows play state, sample position, tempo, time signature, or loop range; `liveTransportForBlock()` builds a daemon-valid block transport object and can optionally compensate sample position by reported output latency, and the live Web Audio worker/fallback paths use that compensation automatically
- check each plugin's `editorKinds` before showing editor actions; native editor buttons should require both `editorKinds: ["native-window"]` and `hello.capabilities.nativeEditor`, and hosts should honor `hello.capabilities.security.nativeEditorFileDialogs` / `nativeEditorClipboard` / `nativeEditorDragAndDrop` before surfacing those UI affordances
- apply only daemon-listed preset ids and pass native-approved preset/sample/cache/license files through opaque session-owned file grants attached to the owning plugin instance; check each plugin's `fileGrantOperations` before showing file-backed actions, use `useFileGrant restoreState` / `loadPreset` / `saveStateDirectory` for advertised worker-native preset/state files, and avoid arbitrary filesystem access for remaining file workflows
- degrade gracefully when the daemon disconnects

## DJ And Live Performance Effects

For a DJ app or live-performance tool, treat SoundBridge as an out-of-process effects rack rather than as a UI-first plugin panel. The host should own deck routing, metering, beat/grid timing, wet/dry controls, bypass, and emergency failover. SoundBridge should own plugin scanning, instance lifecycle, bounded parameter/state/preset operations, and native worker crash containment.

The web client exports `SoundBridgeLiveEffectRack` as a small host-side pattern for this mode:

- create one rack per deck, bus, send, or master insert
- use `createLivePerformanceRackChain()` when a deck, send, or master insert owns a bounded serial chain of racks; it applies block-based live defaults, pipes stage output forward, reports aggregate latency/tail health plus stage aggregate health, per-stage health, per-stage duration, chain dry-output count/reason, and chain deadline lead/jitter, exposes `chain.health`, `chain.timing`, `chain-response-deadline-missed`, `chain-process-timeout-tripped`, and chain latency/budget/timeout/retry/recovery events, can be manually bypassed for deck-level dry output, can apply bounded wet/dry transition fades, skips stale scheduled blocks, fails dry on aggregate chain timeout, fails dry on repeated chain budget misses until `retry()` or an optional bounded dry cooldown clears the recoverable pressure, can auto-recover chain timeouts after a bounded dry cooldown, and fails dry to the last known audio if a stage throws
- process stereo main-bus blocks through binary `processBlock()` by default when the plugin is healthy
- use `setWetMix()` or per-block `wetMix` for host-side dry/wet control without treating mix-zero as a plugin failure; on serial chains, use `chain.setWetMix()` or `chain.processBlock(..., { wetMix })` as a deck/send/master macro while keeping `stageWetMixes` for individual plugin slots
- drive parameters, block automation events/curves, stored timeline lanes, presets, and MIDI through rack helpers bound to the owned instance
- use `createLiveEffectRackPolicy()` before instantiation, `createLiveEffectRackCalibrationWindow()` to record `rack.health` during soundcheck or a set, `createLiveEffectRackChainCalibrationWindow()` to record aggregate chain health and timeout pressure, `createLiveEffectRackFrameBatchCalibrationWindow()` to record shared-frame deck/send/master batch health, `liveEffectRackPolicyOptionsFromCalibration()`, `rack.setTimingPolicy()`, `chain.setTimingPolicy()`, `frameBatch.setTimingPolicy()`, `scheduler.setTimingPolicy()`, and `refreshLiveEffectRackLatencyFromCalibration()` to apply bounded recommendations without recreating healthy processors, `calibrateLiveEffectRackPolicy()` for direct timing samples, and `rack.timing`, `chain.timing`, or `frameBatch.timing` after creation for block duration, latency blocks, freshness windows, and deadline budgets in host scheduling units; cumulative windows baseline dry-output, deadline, and timeout counters on the first recorded health sample, render-timeout pressure can widen bounded timeout recommendations when the host chooses compatibility over strict fail-dry, and frame-batch calibration treats each dry, skipped, failed, or tripped batch frame as current pressure
- use `createLiveEffectRackAdaptiveLatencyController()` from a non-audio control loop when a DJ host wants bounded rack-local transport-latency steps after sustained deadline or jitter pressure on one rack; it observes `rack.health`, waits for a minimum sample window, applies capped increases through `refreshLatency()`, cools down between changes, and can recover downward in bounded steps after stable health without dropping below the configured floor
- use `createLiveEffectRackSchedulerAdaptiveLatencyController()` when a single host-owned effect is driven by `createLiveEffectRackBlockScheduler()`; it records rack health, applies capped scheduler latency increases, keeps scheduler deadline pressure observable, and recovers downward after a stable rack-health window
- use `createLiveEffectRackChainSchedulerAdaptiveLatencyController()` when a deck, send, or master chain is driven by `createLiveEffectRackBlockScheduler()`; it records chain health, applies capped scheduler latency increases from chain calibration, keeps scheduler deadline pressure observable, and recovers downward after a stable chain-health window
- use `createLiveEffectRackFrameBatchSchedulerAdaptiveLatencyController()` when several deck, send, or master targets share one scheduled frame; it records frame-batch health, applies capped scheduler latency increases from aggregate batch calibration, keeps batch dry-pressure reasons observable, and recovers downward after stable batch frames
- use `createLiveEffectRackBlockScheduler()` to prepare host-owned live blocks with monotonic block/sample positions, capture timestamps, bounded stale-capture checks, deadline/process-pressure snapshots, and latency-compensated transport before passing scheduled blocks to `rack.processScheduledBlock()` or `chain.processScheduledBlock()`; use `scheduler.updateFromRackHealth()` for a single rack's full reported output latency and deadline lead/jitter, `scheduler.updateFromChainHealth()` for a serial chain's aggregate latency and deadline lead/jitter, `scheduler.setTimingPolicy()` when soundcheck or host policy changes the stale-input window or deadline/jitter pressure thresholds, `scheduler.updateFromChainCalibration()` after soundcheck to add calibrated transport headroom and expose process-budget/timeout pressure reasons, and `scheduler.updateFromFrameBatchHealth()` or `scheduler.updateFromFrameBatchCalibration()` when several deck/send/master targets share one scheduler frame; pass `skipOnDeadlinePressure: true` only when the host wants an explicit dry skip instead of processing a block already flagged by the scheduler, and read the attached pressure snapshot from the dry response or event detail
- listen for rack `response-deadline-missed`, `process-budget-tripped`, `render-budget-tripped`, `process-timeout`, and `process-timeout-tripped` events when live processing exceeds the current deadline, process/render budget streaks, browser request timeouts, daemon `render_timeout`, or daemon `render_quarantined` failures need host policy; deadline misses can stay wet for adaptive latency, while trip and timeout events mark fail-dry paths and ordinary plugin errors still arrive through `effect-error`
- call `scheduler.captureFrame()` once per host audio quantum and `scheduler.scheduleFromFrame()` for each deck, send, or master target that must share the same block id, sample position, freshness verdict, deadline-pressure snapshot, and latency-compensated transport
- use `createLivePerformanceFrameBatchProcessor()` with `frame-batch-dry-output`, `frame-batch-response-deadline-missed`, `createLiveEffectRackFrameBatchCalibrationWindow()` and `createLiveEffectRackFrameBatchSchedulerAdaptiveLatencyController()` when a host wants to process several scheduled racks or chains from one shared frame with block-derived live budget/timeout defaults, collect bounded aggregate dry, failure, duration, deadline lead/jitter, measured/reported latency, health, dry-output reasons, timeout target attribution, budget/timeout events, soundcheck recommendations, and shared-scheduler latency adjustments for that audio quantum, optionally fail the batch dry after repeated aggregate process-budget misses until `retry()` or a bounded dry recovery window clears the trip, and fail dry immediately when the aggregate frame misses `processTimeoutMs`
- use `createLivePerformanceAudioNodePolicy()` before creating Web Audio graph effects, `createLivePerformanceAudioNodeCalibrationWindow()` after collecting `SoundBridgeAudioNode.health` render duration, response jitter, deadline, underrun, shared-ring current/peak pressure, shared-worker in-flight saturation, and shared-buffer allocation samples, and `livePerformanceAudioNodeOptionsFromCalibration()` / `refreshLivePerformanceAudioNodeLatencyFromCalibration()` to apply bounded output-latency, shared-ring, timeout, and transport-latency recommendations; the window baselines cumulative drop and allocation counters on the first recorded health sample
- use `createLivePerformanceAudioNodeAdaptiveLatencyController()` from a non-audio control loop when a Web Audio graph effect should automatically apply capped transport-latency refreshes after sustained AudioNode pressure and recover downward after stable health; `SoundBridgeAudioNode.refreshLatency(targetSamples)` retargets the worklet's bounded output latency in place when the target maps to a different block count, and controller snapshots still set `recreateRecommended` plus bounded current/recommended values when soundcheck/live measurements call for fixed worklet options such as larger shared rings, higher max output latency, or longer request timeouts that cannot be changed in place
- use `createLivePerformanceAudioNodeRecoveryController()` from a non-audio control loop when a host wants bounded dry-cooldown retries after transport-pressure or render-budget auto-bypass; audio-error recovery is opt-in because retrying unknown plugin failures should be a host decision
- use `createLivePerformanceAudioNodeRecreateController()` from a non-audio control loop when a Web Audio graph host wants one bounded dry-cooldown recreation attempt after `process-timeout`; the host supplies the async function that destroys the quarantined daemon instance, creates a fresh instance/node, and reconnects the graph
- use `liveTransportForBlock()` to build bounded per-block transport for tempo-synced AU/VST effects and stored automation lanes; if no explicit transport is supplied, the live rack generates a bounded block transport with host/output-latency compensation
- return dry audio immediately when the rack is manually bypassed
- optionally fade the first samples of wet/dry transitions to reduce bypass clicks
- return dry audio for timestamped input that is already older than the host's configured live window
- return dry audio instead of queuing new work when the rack's in-flight block limit is full
- fail dry on a bounded `processBlock()` timeout when the host configures one
- return dry audio and mark the rack unhealthy when `processAudioBlock` fails
- count end-to-end host-to-bridge processing budget misses and fail dry after a bounded consecutive miss threshold
- count render-budget misses and fail closed to dry audio after a bounded consecutive miss threshold
- optionally recover from render-pressure trips after a bounded number of dry cooldown blocks
- optionally recover from end-to-end processing pressure after a bounded number of dry cooldown blocks
- call `retry()` from a UI/control path after recoverable process-budget or render-budget pressure without recreating the plugin
- optionally recreate once after a process-timeout cooldown for live hosts that prefer automatic fail-dry recovery
- call `recreate()` from a non-audio-control path when the UI/user decides to retry a plugin after non-recoverable errors or exhausted recovery attempts; rack `health.processTimeoutRecoveryExhausted` and `process-timeout-recovery-exhausted` mark process-timeout cases where no further automatic timeout recreation will run, and rack recreate emits `recreate-started`, `recreated`, or `recreate-failed` so host UI can show replacement state without polling

This is the right policy for live sets: a stale, slow, failed, backed-up, timed-out, or repeatedly over-budget effect should become a dry bypass, not silence and not an app crash. Rack `wetMix` is bounded to `0..1`; mix-zero can still process the plugin for state continuity, while `setBypassed(true)` is the explicit CPU-saving and emergency fail-dry path. For serial chains, `chain.setWetMix()` blends the whole processed chain against the original dry input, while `chain.setBypassed(true)` skips every stage and returns dry audio for the whole deck, send, or master chain while preserving the same wet/dry transition fade behavior. Chain `health` exposes sample-rate-aware aggregate latency, tail, infinite-tail state, `stageHealthy`, `failedStageIndex`, `stageResults`, cumulative `dryOutputBlocks`, `bypassDryOutputBlocks`, `lastDryReason`, process deadline lead, response jitter, deadline misses, timeout stage attribution, timeout recovery cooldown, automatic timeout recovery attempts, timeout recovery exhaustion, and the latest stage error, `chain.timing` reports aggregate latency and budget values in sample, millisecond, and block units, and `createLiveEffectRackChainCalibrationWindow()` converts chain health into bounded readiness, process-budget, timeout, dry-output, deadline, jitter, and aggregate-latency recommendations for deck-level soundcheck. Frame batch timeout results and events preserve completed target results while marking pending targets dry, and frame batch health exposes `recoveryDryBlocksRemaining`, `processTimeoutRecoveryAttempts`, `processTimeoutRecoveryExhausted`, and measured plus reported latency so shared-frame schedulers can compensate with the larger safe value. The chain emits `latencychange` or `healthchange` when those values change, racks and chains emit `dry-output` for every dry block with `{ response, health, reason }`, chains emit `chain-process-timeout-recovery-exhausted`, and frame batches emit `frame-batch-dry-output` plus `frame-batch-process-timeout-recovery-exhausted` so deck-level hosts can compensate and display live fail-dry behavior without polling every plugin slot. Rack control helpers target the rack-owned instance and reject when the rack is unhealthy, while manual bypass still permits live knob and preset changes. `transitionFadeSamples` defaults to `0`; live hosts can set a small bounded value to smooth wet/dry edge changes when bypass, stale input, backpressure, timeout, process-pressure recovery, or render-pressure recovery changes the rack's output path. `maxInputAgeMs` defaults to `0`; live hosts can set it when they pass `performance.now()`-style input timestamps to avoid spending native processing on blocks that already missed the host's live window. `maxInFlightBlocks` defaults to `1` so live rack callers cannot accidentally queue an unbounded backlog behind a slow plugin, and the SDK bounds main and input-bus blocks to the rack's configured channel counts and `maxBlockSize`, pads ragged channels, and replaces non-finite samples before binary or JSON transport. `processBudgetMs` defaults to `0` on ordinary racks; live hosts can enable it or use `createLivePerformance()` to track total host-to-bridge block time and fail dry after bounded repeated misses. `processTimeoutMs` defaults to `0` so compatibility hosts can keep normal request-timeout behavior; live hosts can set a small bounded timeout from their block size and monitoring latency policy, and positive values bound browser request lifetime, worker-side late-response delivery, aggregate chain and frame-batch processing, and the daemon's per-render worker command deadline. `createLiveEffectRackCalibrationWindow()`, `createLivePerformanceAudioNodeCalibrationWindow()`, `calibrateLiveEffectRackPolicy()`, and `calibrateLivePerformanceAudioNodePolicy()` accept bounded live timing samples, then return clamped latency, timeout, readiness, and warning recommendations for the host's scheduling UI without ratcheting budgets when the current policy already has the requested safety margin; rack and AudioNode calibration snapshots also include recommended options for the next rack/node plus helpers to refresh current transport latency from the recommendation. Live rack `health` exposes bounded process-budget, deadline-lead, response-jitter, deadline-miss, timeout, block-budget, timeout-minus-budget, timeout-count, quarantine, cumulative `dryOutputBlocks`, `lastDryReason`, and `recoveryDryBlocksRemaining` fields from those paths so UI and recovery policy can distinguish end-to-end bridge pressure, stale input/output, backpressure, plugin render pressure, and generic browser request timeouts; changes to `lastDryReason` also emit `healthchange`. `maxConsecutiveRenderBudgetMisses` defaults to `3`, can be set to `0` to observe without failing closed, and is capped to keep host policy bounded. `SoundBridgeAudioNode.createLivePerformance()` uses a default render-budget threshold of `2`, a default sustained transport-pressure threshold of `3`, fails dry on the first audio error, updates plugin latency from render diagnostics, exposes fallback-output stats and bounded `fallback-output` events through `fallbackOutputBlocks`, `lastFallbackReason`, and `{ deltaBlocks, reason, stats, health }`, exposes policy/calibration helpers for graph-level scheduling, and emits `transport-pressure-auto-bypassed`, `render-budget-tripped`, `render-budget-auto-bypassed`, `process-timeout`, `process-timeout-tripped`, `process-timeout-auto-bypassed`, or `audio-error-auto-bypassed`; hosts retry ordinary pressure with `setBypassed(false)` but recreate the node for `process-timeout`, optionally using `createLivePerformanceAudioNodeRecreateController()` to wait through bounded dry blocks and cap recreate attempts. `processBudgetRecoveryBlocks`, `renderBudgetRecoveryBlocks`, `processTimeoutRecoveryBlocks`, and `maxProcessTimeoutRecoveries` default to `0` on ordinary racks; `createLivePerformance()` sets bounded dry-cooldown recovery defaults for process pressure, render pressure, and one process-timeout recreation attempt. ## Current Prototype Limitations

- Main-bus and bus-indexed audio blocks use binary WebSocket frames in the reference web client. The demo opts into worker-owned WebSocket transport with a direct `AudioWorklet` audio port and `SharedArrayBuffer` rings when cross-origin isolation is available. The shared path wakes the transport worker with `Atomics.waitAsync`/`notify` where supported, falls back to a 1 ms timer poll when atomic waits are unavailable, pauses shared pumps while disconnected, and reports whether the shared path is using atomic or timer wakeups. The worklet reports native render duration, render-budget misses, response deadline lead, jitter in both render blocks and samples, and current plus peak shared-ring queue depth, so hosts can tell plugin-side render pressure from transport timing. The fallback port path bounds in-flight posts at the worklet edge, and positive `audioRequestTimeoutMs` values bound how long worker-owned and page-fallback browser audio requests wait before failing dry and suppressing stale late responses; `0` disables the browser-side response timer. Recycled input buffers after worker send, transferred binary output buffers without an extra clone, bounded adaptive latency changes, controlled safety blocks when underruns, shared-output pressure, or deadline pressure force a latency raise, and transport latency samples keep the path observable. JSON audio blocks remain the compatibility path for hosts that opt out of binary transport.
- Parameter automation supports bounded event lists, bounded per-block step/linear curves with sample offsets, and stored absolute-sample timeline lanes applied from `processAudioBlock.transport.samplePosition`.
- Editor support currently means bounded generic parameter editor sessions plus an opt-in native editor broker contract; platform-specific native plugin windows remain future broker implementation work.
- The mock plugin is a gain effect.
- The demo uses a `MessagePort` queue between main thread and AudioWorklet for compatibility.
- Installed VST3 audio effects can instantiate, render, expose path-free scanner metadata, expose parameters, expose bounded SDK unit metadata, expose bounded plugin-level and parameter-attached program-list metadata, export and restore bounded opaque SDK program data for daemon-listed programs when supported, expose bounded note-expression metadata, apply bounded listed preset snapshots, load bounded worker-native preset-state snapshots through file grants, set parameters, accept bounded MIDI note/poly-pressure events, mapped MIDI controllers, and value/text note-expression events, accept bounded host transport context, report bounded per-bus layout metadata, negotiate bounded per-bus SDK speaker arrangements with main-bus fallback, process explicit bus-indexed input buffers, return bus-indexed output buffers, report bounded plugin latency and tail time, and save/restore opaque state when the Steinberg SDK worker is linked. Realtime-compatible installed Audio Units can do the same through the CoreAudio worker, including path-free AudioComponent identifiers, bounded host transport callbacks, bounded short MIDI messages where the unit supports them, active CoreAudio input-element routing from explicit bus-indexed buffers, active CoreAudio output-element rendering into explicit bus-indexed buffers, and explicit realtime format-converter, multi-source merger, and multi-output splitter profile metadata for supported Apple utility units; AU offline effects and incompatible utility profiles remain discovery-only or roadmap items for now. Compatible installed LV2 audio/control effects can instantiate, render, expose path-free LV2 URI and block-size profile metadata, expose TTL-derived control ports as parameters, apply bounded listed preset snapshots, load bounded worker-native preset-state snapshots through file grants, set parameters, save/restore bounded control-port state, portable POD extension state, and brokered file-backed state, deliver bounded atom MIDI and LV2 atom `time:Position` transport to compatible input ports, expose bounded `buf-size:boundedBlockLength`, `fixedBlockLength`, `powerOf2BlockLength`, and `options#options` host data, run bounded synchronous LV2 worker jobs for plugins that expose `work:interface`, report bounded standard LV2 latency output ports, report bounded LV2 `pg:group` main buses with aggregate/per-audio-port fallback for ungrouped metadata, route explicit bus-indexed input buffers, return bus-indexed output buffers, and report conservative tail metadata through the basic LV2 worker.
- The website-playable VST3/AU/LV2 instruments are repo-local example bundles rendered by the native example renderer; LV2 UI declaration metadata is path-free scanner data, and grant-backed worker-native preset/state load/restore/save works through `useFileGrant`. LV2 UI hosting, platform-native plugin UI, advanced LV2 extension profiles, and vendor-specific sample/cache/license file handlers remain security-sensitive native milestones.
