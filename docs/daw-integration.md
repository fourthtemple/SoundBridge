# Web DAW Integration

SoundBridge should feel like an AudioNode plus a plugin-management API.

## Minimal Host Flow

1. Create `AudioContext`.
2. Create `SoundBridgeClient` with `ws://127.0.0.1:47370/bridge`.
3. Call `connect()`, `hello()`, and `pair()`.
4. Call `scanPlugins()` or `listPlugins()`.
5. Create a plugin instance with the current sample rate and block size.
6. Create `SoundBridgeAudioNode`.
7. Connect source nodes into the bridge node and connect the bridge node to the destination.
8. For instruments, send notes or MIDI clips through `sendMidiEvents()`; for VST3 note expression, include explicit bounded `noteId` values.
9. Apply bounded listed presets with `setPreset()` when the selected plugin exposes preset metadata.
10. Check the plugin's `editorKinds`, then open a bounded generic parameter editor with `openEditor()` or a configured native editor broker when both the plugin and daemon advertise it.
11. Close editor sessions with `closeEditor()` when the UI tab or panel is done.
12. Store `getState()` output in the DAW project.
13. Restore using `setState()` when reopening the project.

## WebAudioModules Compatibility

The first compatibility target is an adapter that exposes:

- descriptor metadata from `listPlugins()`
- format-aware plugin identity for VST3, AU, LV2, and mock plugins
- an AudioNode-like instance backed by `SoundBridgeAudioNode`
- normalized parameter automation
- bounded preset snapshot application
- opaque state serialization
- generic parameter UI fallback

The adapter should not require hosts to adopt SoundBridge-specific UI primitives.

## Host Responsibilities

Hosts should:

- clearly show when audio is being routed through a local daemon
- use `createInstance().layout` or `getLayout()` when sizing host nodes and meters
- store opaque plugin state without editing it
- account for `getLatency()` in timeline scheduling and monitoring UI; `SoundBridgeLiveEffectRack.refreshLatency()`, `SoundBridgeAudioNode.refreshLatency()`, `createLivePerformanceAudioNodePolicy()`, `createLivePerformanceAudioNodeCalibrationWindow()`, `calibrateLivePerformanceAudioNodePolicy()`, and render-response diagnostics keep plugin, transport, and total reported latency in host-readable sample and millisecond health
- call `SoundBridgeAudioNode.setBypassed(true)` for Web Audio graph emergency fail-dry paths instead of destroying and recreating the plugin instance during a live set; the worklet outputs dry input and discards queued or pending wet responses from before the bypass transition, and `setBypassed(false)` is the explicit retry after transport-pressure, render-pressure, or audio-error auto-bypass
- read `SoundBridgeAudioNode.health` and listen for `latencychange`, `transport-pressure`, `transport-pressure-auto-bypassed`, `render-budget-exceeded`, `render-budget-auto-bypassed`, `audio-error-auto-bypassed`, and `audio-error` when using the Web Audio graph path; health includes bypass state, transport pressure, transport-pressure auto-bypass, render pressure, render-budget auto-bypass, audio errors, audio-error auto-bypass, plugin/transport/reported latency in samples and milliseconds, render-response plugin latency changes, adaptive latency counters, deadline misses, stale output blocks, dropped input blocks, underruns, shared-audio drops, response jitter against the configured threshold, and the latest pressure/latency reasons; shared-ring input pressure and response jitter can raise bounded adaptive latency, and `response-jitter` can appear as a transport-pressure reason when a high-jitter window triggers a latency increase
- send bounded `processAudioBlock.transport` context when the host knows play state, sample position, tempo, time signature, or loop range; `liveTransportForBlock()` builds a daemon-valid block transport object and can optionally compensate sample position by reported output latency, and the live Web Audio worker/fallback paths use that compensation automatically
- check each plugin's `editorKinds` before showing editor actions; native editor buttons should require both `editorKinds: ["native-window"]` and `hello.capabilities.nativeEditor`, and hosts should honor `hello.capabilities.security.nativeEditorFileDialogs` / `nativeEditorClipboard` / `nativeEditorDragAndDrop` before surfacing those UI affordances
- apply only daemon-listed preset ids and pass native-approved preset/sample/cache/license files through opaque session-owned file grants attached to the owning plugin instance; check each plugin's `fileGrantOperations` before showing file-backed actions, use `useFileGrant restoreState` / `loadPreset` / `saveStateDirectory` for advertised worker-native preset/state files, and avoid arbitrary filesystem access for remaining file workflows
- degrade gracefully when the daemon disconnects

## DJ And Live Performance Effects

For a DJ app or live-performance tool, treat SoundBridge as an out-of-process effects rack rather than as a UI-first plugin panel. The host should own deck routing, metering, beat/grid timing, wet/dry controls, bypass, and emergency failover. SoundBridge should own plugin scanning, instance lifecycle, bounded parameter/state/preset operations, and native worker crash containment.

The web client exports `SoundBridgeLiveEffectRack` as a small host-side pattern for this mode:

- create one rack per deck, bus, send, or master insert
- use `createLivePerformanceRackChain()` when a deck, send, or master insert owns a bounded serial chain of racks; it applies block-based live defaults, pipes stage output forward, reports aggregate latency/tail health plus stage aggregate health, per-stage health, per-stage duration, and chain deadline lead/jitter, exposes `chain.health`, `chain.timing`, and chain latency/budget/retry/recovery events, can be manually bypassed for deck-level dry output, can apply bounded wet/dry transition fades, skips stale scheduled blocks, fails dry on repeated chain budget misses until `retry()` or an optional bounded dry cooldown clears the recoverable pressure, and fails dry to the last known audio if a stage throws
- process stereo main-bus blocks through binary `processBlock()` by default when the plugin is healthy
- use `setWetMix()` or per-block `wetMix` for host-side dry/wet control without treating mix-zero as a plugin failure; on serial chains, use `chain.setWetMix()` or `chain.processBlock(..., { wetMix })` as a deck/send/master macro while keeping `stageWetMixes` for individual plugin slots
- drive parameters, block automation events/curves, stored timeline lanes, presets, and MIDI through rack helpers bound to the owned instance
- use `createLiveEffectRackPolicy()` before instantiation, `createLiveEffectRackCalibrationWindow()` to record `rack.health` during soundcheck or a set, `createLiveEffectRackChainCalibrationWindow()` to record aggregate chain health, `liveEffectRackPolicyOptionsFromCalibration()` and `refreshLiveEffectRackLatencyFromCalibration()` to apply bounded recommendations, `calibrateLiveEffectRackPolicy()` for direct timing samples, and `rack.timing` or `chain.timing` after creation for block duration, latency blocks, freshness windows, and deadline budgets in host scheduling units; the window baselines cumulative dry-output, deadline, and timeout counters on the first recorded health sample
- use `createLiveEffectRackAdaptiveLatencyController()` from a non-audio control loop when a DJ host wants bounded transport-latency steps after sustained deadline or jitter pressure on one rack; it observes `rack.health`, waits for a minimum sample window, applies capped increases through `refreshLatency()`, cools down between changes, and can recover downward in bounded steps after stable health without dropping below the configured floor
- use `createLiveEffectRackChainSchedulerAdaptiveLatencyController()` when a deck, send, or master chain is driven by `createLiveEffectRackBlockScheduler()`; it records chain health, applies capped scheduler latency increases from chain calibration, keeps scheduler deadline pressure observable, and recovers downward after a stable chain-health window
- use `createLiveEffectRackBlockScheduler()` to prepare host-owned live blocks with monotonic block/sample positions, capture timestamps, bounded stale-capture checks, deadline-pressure snapshots, and latency-compensated transport before passing `scheduled.request` to `rack.processBlock()` or `chain.processScheduledBlock()`; use `scheduler.updateFromRackHealth()` for a single rack's output latency and deadline lead/jitter, `scheduler.updateFromChainHealth()` for a serial chain's aggregate latency and deadline lead/jitter, or `scheduler.updateFromChainCalibration()` after soundcheck to add calibrated transport headroom to the current chain latency
- use `createLivePerformanceAudioNodePolicy()` before creating Web Audio graph effects, `createLivePerformanceAudioNodeCalibrationWindow()` after collecting `SoundBridgeAudioNode.health` render duration, response jitter, deadline, underrun, and shared-ring pressure samples, and `livePerformanceAudioNodeOptionsFromCalibration()` / `refreshLivePerformanceAudioNodeLatencyFromCalibration()` to apply bounded output-latency, shared-ring, timeout, and transport-latency recommendations; the window baselines cumulative drop counters on the first recorded health sample
- use `liveTransportForBlock()` to build bounded per-block transport for tempo-synced AU/VST effects and stored automation lanes; if no explicit transport is supplied, the live rack generates a bounded block transport with host/output-latency compensation
- return dry audio immediately when the rack is manually bypassed
- optionally fade the first samples of wet/dry transitions to reduce bypass clicks
- return dry audio for timestamped input that is already older than the host's configured live window
- return dry audio instead of queuing new work when the rack's in-flight block limit is full
- fail dry on a bounded `processBlock()` timeout when the host configures one
- return dry audio and mark the rack unhealthy when `processAudioBlock` fails
- count end-to-end host-to-bridge processing budget misses and fail dry after a bounded consecutive miss threshold
- count render-budget misses and fail closed to dry audio after a bounded consecutive miss threshold
- optionally recover from render-pressure trips after a bounded number of dry cooldown blocks
- optionally recover from end-to-end processing pressure after a bounded number of dry cooldown blocks
- call `retry()` from a UI/control path after recoverable process-budget or render-budget pressure without recreating the plugin
- optionally recreate once after a process-timeout cooldown for live hosts that prefer automatic fail-dry recovery
- call `recreate()` from a non-audio-control path when the UI/user decides to retry a plugin after non-recoverable errors or exhausted recovery attempts

This is the right policy for live sets: a stale, slow, failed, backed-up, or repeatedly over-budget effect should become a dry bypass, not silence and not an app crash. Rack `wetMix` is bounded to `0..1`; mix-zero can still process the plugin for state continuity, while `setBypassed(true)` is the explicit CPU-saving and emergency fail-dry path. For serial chains, `chain.setWetMix()` blends the whole processed chain against the original dry input, while `chain.setBypassed(true)` skips every stage and returns dry audio for the whole deck, send, or master chain while preserving the same wet/dry transition fade behavior. Chain `health` exposes sample-rate-aware aggregate latency, tail, infinite-tail state, `stageHealthy`, `failedStageIndex`, `stageResults`, `lastDryReason`, process deadline lead, response jitter, deadline misses, and the latest stage error, `chain.timing` reports aggregate latency and budget values in sample, millisecond, and block units, and `createLiveEffectRackChainCalibrationWindow()` converts chain health into bounded readiness, process-budget, dry-output, deadline, jitter, and aggregate-latency recommendations for deck-level soundcheck. The chain emits `latencychange` or `healthchange` when those values change, so deck-level hosts can compensate and display the whole chain without polling every plugin slot. Rack control helpers target the rack-owned instance and reject when the rack is unhealthy, while manual bypass still permits live knob and preset changes. `transitionFadeSamples` defaults to `0`; live hosts can set a small bounded value to smooth wet/dry edge changes when bypass, stale input, backpressure, timeout, process-pressure recovery, or render-pressure recovery changes the rack's output path. `maxInputAgeMs` defaults to `0`; live hosts can set it when they pass `performance.now()`-style input timestamps to avoid spending native processing on blocks that already missed the host's live window. `maxInFlightBlocks` defaults to `1` so live rack callers cannot accidentally queue an unbounded backlog behind a slow plugin, and the SDK bounds main and input-bus blocks to the rack's configured channel counts and `maxBlockSize`, pads ragged channels, and replaces non-finite samples before binary or JSON transport. `processBudgetMs` defaults to `0` on ordinary racks; live hosts can enable it or use `createLivePerformance()` to track total host-to-bridge block time and fail dry after bounded repeated misses. `processTimeoutMs` defaults to `0` so compatibility hosts can keep normal request-timeout behavior; live hosts can set a small bounded timeout from their block size and monitoring latency policy, and positive values bound browser request lifetime, worker-side late-response delivery, and the daemon's per-render worker command deadline. `createLiveEffectRackCalibrationWindow()`, `createLivePerformanceAudioNodeCalibrationWindow()`, `calibrateLiveEffectRackPolicy()`, and `calibrateLivePerformanceAudioNodePolicy()` accept bounded live timing samples, then return clamped latency, timeout, readiness, and warning recommendations for the host's scheduling UI without ratcheting budgets when the current policy already has the requested safety margin; rack and AudioNode calibration snapshots also include recommended options for the next rack/node plus helpers to refresh current transport latency from the recommendation. A missed native render deadline terminates and quarantines that worker process, returns a bounded `render_timeout` protocol error, then fails fast as `render_quarantined` until the host recreates the instance; hosts should use it for live fail-dry paths, not ordinary compatibility playback. Live rack `health` exposes bounded process-budget, deadline-lead, response-jitter, deadline-miss, timeout, block-budget, timeout-minus-budget, timeout-count, quarantine, and `lastDryReason` fields from those paths so UI and recovery policy can distinguish end-to-end bridge pressure, stale input/output, backpressure, plugin render pressure, and generic browser request timeouts; changes to `lastDryReason` also emit `healthchange`. `maxConsecutiveRenderBudgetMisses` defaults to `3`, can be set to `0` to observe without failing closed, and is capped to keep host policy bounded. `SoundBridgeAudioNode.createLivePerformance()` uses a default render-budget threshold of `2`, a default sustained transport-pressure threshold of `3`, fails dry on the first audio error, updates plugin latency from render diagnostics, exposes policy/calibration helpers for graph-level scheduling, and emits `transport-pressure-auto-bypassed`, `render-budget-auto-bypassed`, or `audio-error-auto-bypassed`; hosts retry with `setBypassed(false)`. `processBudgetRecoveryBlocks`, `renderBudgetRecoveryBlocks`, `processTimeoutRecoveryBlocks`, and `maxProcessTimeoutRecoveries` default to `0` on ordinary racks; `createLivePerformance()` sets bounded dry-cooldown recovery defaults for process pressure, render pressure, and one process-timeout recreation attempt. `retry()` clears recoverable process-budget and render-budget pressure while keeping the same plugin instance alive. Processing errors, render timeouts, quarantined workers, and exhausted process-timeout recovery attempts still require `recreate()`. Binary frames, direct worklet-to-worker audio ports, `SharedArrayBuffer` rings where browser isolation allows them, atomic worker wakeups where supported, bounded shared-path in-flight requests, shared ring depth derived from live in-flight and latency bounds, newest-input and newest-output overwrite under shared ring pressure, recycled worklet buffers plus transport-worker shared-input buffers, measured native render duration, per-block render-budget verdicts, measured response deadline lead/jitter, render-response plugin latency diagnostics, and bounded adaptive output latency reduce page-thread serialization work and smooth short jitter bursts. The worklet can also raise latency before dropouts when responses repeatedly arrive below its configured deadline-lead target or jitter beyond the configured live threshold, then recover toward the host's minimum latency after the configured on-time window. Production DJ use still needs deeper shared-memory tuning, latency calibration, and host scheduling policy.

## Current Prototype Limitations

- Main-bus and bus-indexed audio blocks use binary WebSocket frames in the reference web client. The demo opts into worker-owned WebSocket transport with a direct `AudioWorklet` audio port and `SharedArrayBuffer` rings when cross-origin isolation is available. The shared path wakes the transport worker with `Atomics.waitAsync`/`notify` where supported, falls back to a 1 ms timer poll when atomic waits are unavailable, pauses shared pumps while disconnected, and reports whether the shared path is using atomic or timer wakeups. The worklet reports native render duration, render-budget misses, response deadline lead, and jitter in both render blocks and samples, so hosts can tell plugin-side render pressure from transport timing. The fallback port path bounds in-flight posts at the worklet edge, and positive `audioRequestTimeoutMs` values bound how long worker-owned and page-fallback browser audio requests wait before failing dry and suppressing stale late responses; `0` disables the browser-side response timer. Recycled input buffers after worker send, transferred binary output buffers without an extra clone, bounded adaptive latency changes, controlled safety blocks when underruns, shared-output pressure, or deadline pressure force a latency raise, and transport latency samples keep the path observable. JSON audio blocks remain the compatibility path for hosts that opt out of binary transport.
- Parameter automation supports bounded event lists, bounded per-block step/linear curves with sample offsets, and stored absolute-sample timeline lanes applied from `processAudioBlock.transport.samplePosition`.
- Editor support currently means bounded generic parameter editor sessions plus an opt-in native editor broker contract; platform-specific native plugin windows remain future broker implementation work.
- The mock plugin is a gain effect.
- The demo uses a `MessagePort` queue between main thread and AudioWorklet for compatibility.
- Installed VST3 audio effects can instantiate, render, expose path-free scanner metadata, expose parameters, expose bounded SDK unit metadata, expose bounded plugin-level and parameter-attached program-list metadata, export and restore bounded opaque SDK program data for daemon-listed programs when supported, expose bounded note-expression metadata, apply bounded listed preset snapshots, load bounded worker-native preset-state snapshots through file grants, set parameters, accept bounded MIDI note/poly-pressure events, mapped MIDI controllers, and value/text note-expression events, accept bounded host transport context, report bounded per-bus layout metadata, negotiate bounded per-bus SDK speaker arrangements with main-bus fallback, process explicit bus-indexed input buffers, return bus-indexed output buffers, report bounded plugin latency and tail time, and save/restore opaque state when the Steinberg SDK worker is linked. Realtime-compatible installed Audio Units can do the same through the CoreAudio worker, including path-free AudioComponent identifiers, bounded host transport callbacks, bounded short MIDI messages where the unit supports them, active CoreAudio input-element routing from explicit bus-indexed buffers, active CoreAudio output-element rendering into explicit bus-indexed buffers, and explicit realtime format-converter, multi-source merger, and multi-output splitter profile metadata for supported Apple utility units; AU offline effects and incompatible utility profiles remain discovery-only or roadmap items for now. Compatible installed LV2 audio/control effects can instantiate, render, expose path-free LV2 URI and block-size profile metadata, expose TTL-derived control ports as parameters, apply bounded listed preset snapshots, load bounded worker-native preset-state snapshots through file grants, set parameters, save/restore bounded control-port state, portable POD extension state, and brokered file-backed state, deliver bounded atom MIDI and LV2 atom `time:Position` transport to compatible input ports, expose bounded `buf-size:boundedBlockLength`, `fixedBlockLength`, `powerOf2BlockLength`, and `options#options` host data, run bounded synchronous LV2 worker jobs for plugins that expose `work:interface`, report bounded standard LV2 latency output ports, report bounded LV2 `pg:group` main buses with aggregate/per-audio-port fallback for ungrouped metadata, route explicit bus-indexed input buffers, return bus-indexed output buffers, and report conservative tail metadata through the basic LV2 worker.
- The website-playable VST3/AU/LV2 instruments are repo-local example bundles rendered by the native example renderer; LV2 UI declaration metadata is path-free scanner data, and grant-backed worker-native preset/state load/restore/save works through `useFileGrant`. LV2 UI hosting, platform-native plugin UI, advanced LV2 extension profiles, and vendor-specific sample/cache/license file handlers remain security-sensitive native milestones.
